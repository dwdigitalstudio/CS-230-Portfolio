# CS 230 Portfolio Reflection

## The Gaming Room Client and Software Requirements

The client for this project was The Gaming Room. They wanted to expand their existing Android game, *Draw It or Lose It*, into a web-based application that could work across multiple platforms. The software needed to support multiple games, teams, and players while making sure each game and team name was unique. The design also needed to consider scalability, security, memory management, operating platforms, and how the application would support users connecting from different devices.

## What I Did Well

One thing I did particularly well in developing this documentation was explaining the structure of the application in a clear way. I connected the client’s needs to specific design choices, such as using object-oriented programming, the singleton pattern, and separate classes for games, teams, and players. I also did well organizing the document so the client and development team could understand the purpose of the design.

## Helpful Parts of the Design Document Process

Working through the design document was helpful because it made me think about the application before focusing only on the code. It helped me understand what classes were needed, how they related to each other, and why certain design patterns were useful. This made the coding process easier because the project already had a structure and direction.

## What I Would Revise

If I could revise one part of my work, I would improve the recommendations section. I would add more detail to explain why certain choices were best for The Gaming Room, especially the recommendation to use Linux as the server-side operating platform. I would also make the security and database recommendations more specific so the client could better understand how those choices protect users and support future growth.

## Interpreting User Needs

I interpreted the user’s needs by reviewing what The Gaming Room wanted the application to do and turning those needs into software design requirements. Since the client wanted the game to work across different platforms and support multiple users, I focused on a design that was organized, scalable, and reliable. Considering user needs is important because software should solve the client’s actual problem. If the design does not match what the users need, the application may work technically but still fail to be useful.

## My Approach to Software Design

I approached the software design by first identifying the client’s requirements and then breaking the system into smaller parts. I looked at the classes, relationships, design patterns, platform requirements, storage needs, and security concerns before making recommendations. In the future, I would use the same strategy by starting with the client’s goals, creating a clear design plan, comparing technical options, and using object-oriented principles to keep the software organized and maintainable.
